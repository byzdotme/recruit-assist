# Recruit Assist Agent Plugin 设计规格

## 背景与目标

本项目创建一个可在公司 Git 仓库中共享的 agent plugin，协助招聘者完成简历筛选和面试准备。插件首期至少适配 Codex 和 Claude Code：同一份插件源码同时提供 Codex 与 Claude Code 各自需要的 manifest，并共享核心 skills。

首期提供两个 skill：

1. `resume-screening`：读取项目中的 JD 和多份简历，根据 JD 要求生成可解释评分、排序和 Markdown 报告。
2. `interview-prep`：读取 JD、指定候选人简历和面试轮次，为面试官生成考察策略、问题清单、追问方向和评分建议。

输入格式要求：

- JD 主要支持 Markdown 和 TXT。
- 简历主要支持 PDF，兼容 DOCX。
- 输出以 Markdown 文件为主，便于团队共享、评审和版本管理。

## 推荐方案

采用“双 manifest、共享 skills”的插件结构：

```text
recruit-assist/
├── .codex-plugin/
│   └── plugin.json
├── .claude-plugin/
│   └── plugin.json
├── skills/
│   ├── resume-screening/
│   │   ├── SKILL.md
│   │   └── references/
│   │       ├── scoring-rubric.md
│   │       └── report-template.md
│   └── interview-prep/
│       ├── SKILL.md
│       └── references/
│           ├── interview-rounds.md
│           └── question-template.md
└── README.md
```

Codex 使用 `.codex-plugin/plugin.json` 识别插件，Claude Code 使用 `.claude-plugin/plugin.json` 识别插件。两个平台共用 `skills/`，避免重复维护和平台间行为漂移。

## 插件 manifest 设计

Codex manifest 放在 `.codex-plugin/plugin.json`，包含：

- `name`: `recruit-assist`
- `version`: `0.1.0`
- `description`: 简述招聘辅助能力。
- `author`: 使用团队占位信息，可由公司仓库维护者替换。
- `license`: 默认 `MIT`，除非团队后续要求调整。
- `keywords`: `recruiting`、`resume-screening`、`interview-prep`、`hiring`
- `skills`: `./skills/`
- `interface`: 提供展示名、简短描述、长描述、分类和默认提示。

Claude Code manifest 放在 `.claude-plugin/plugin.json`，包含：

- `name`: `recruit-assist`
- `version`: `0.1.0`
- `description`: 与 Codex 保持一致。
- `author`: 与 Codex 保持一致。
- `repository`: 后续由公司 Git 仓库地址替换。
- `license`: 与 Codex 保持一致。
- `keywords`: 与 Codex 保持一致。

Claude Code 的插件规范默认发现 `skills/`，因此首期不额外配置 commands、agents、hooks 或 MCP servers。

## Skill 一：简历筛选

### 触发场景

当用户要求根据 JD 批量筛选、评估、排序简历，或要求生成候选人 shortlist、评分报告、简历匹配分析时触发。

### 输入

- 一个 JD 文件，格式为 Markdown 或 TXT。
- 一个简历目录或多个简历文件，格式主要为 PDF，兼容 DOCX。
- 可选输出路径；未指定时建议输出到当前项目的 `recruit-assist-output/`。

### 工作流

1. 读取 JD，提取岗位职责、硬性要求、加分项、经验年限、技术栈、业务领域、语言和地点等要求。
2. 读取每份简历，抽取候选人姓名、联系方式线索、当前职位、年限、教育、公司经历、项目经历、技能和风险点。
3. 以 JD 为唯一评分基准，不根据与 JD 无关的信息加分。
4. 为每位候选人生成 100 分制可解释评分。
5. 按总分排序，生成 Markdown 报告。
6. 对信息不足、文件解析失败、疑似 OCR 质量差的简历单独标记，不静默忽略。

### 评分模型

默认总分 100 分：

- 硬性要求匹配：30 分。包括必须技能、经验年限、地点、语言、资质等。
- 相关项目和业务经验：25 分。看项目职责、业务领域、复杂度和与 JD 的贴合度。
- 技术能力或岗位能力深度：20 分。按岗位类型评估技术栈、工具链、方法论或专业能力。
- 能力递进与项目连续性：10 分。只评价简历中与岗位能力递进、职责范围扩大、项目连续性和协作线索相关的证据；不得基于年龄、毕业年份、职业空窗原因、家庭状态等作推断。
- 加分项：10 分。仅使用 JD 明确提到的 preferred qualifications 或明显相关亮点。
- 风险和缺口控制：5 分。根据关键缺失、频繁跳槽、信息矛盾、表达模糊等扣分。

报告必须包含：

- 候选人排序表。
- 每位候选人的分项分数和总分。
- 支撑判断的简历证据摘要。
- 主要缺口和风险。
- 建议动作：推荐面试、备选、暂缓或不推荐。
- 后续面试验证问题。

### 输出 Markdown 结构

```text
# 简历筛选报告

## 岗位摘要

## 排名总览

## 候选人明细

## 风险与复核项

## 建议面试名单
```

## Skill 二：面试准备

### 触发场景

当用户要求基于 JD 和某份简历准备面试，或要求按业务面、技术面、HR 面等轮次生成面试问题和考察策略时触发。

### 输入

- 一个 JD 文件，格式为 Markdown 或 TXT。
- 一份指定候选人简历，格式为 PDF 或 DOCX。
- 面试轮次，例如业务面、技术面、HR 面、终面、文化面。
- 可选面试时长；未指定时默认 45 分钟。

### 工作流

1. 提取 JD 的岗位目标、核心能力、硬性要求和加分项。
2. 提取候选人简历中的关键经历、优势、疑点和与 JD 的差距。
3. 根据面试轮次选择考察重点。
4. 生成面试结构，包括开场、核心问题、追问、实操或案例题、候选人提问和收尾。
5. 为每个问题标注考察目的、优秀回答信号、风险信号和建议追问。
6. 输出 Markdown 面试准备文档。

### 轮次策略

- 业务面：重点验证业务理解、过往场景、问题拆解、协作方式和结果质量。
- 技术面：重点验证技术深度、架构判断、工程实践、问题定位和项目真实性。
- HR 面：重点验证动机、稳定性、薪资期望、文化匹配、沟通风格和入职风险。
- 终面：重点验证岗位匹配度、长期潜力、关键风险和录用决策所需信息。

### 输出 Markdown 结构

```text
# 面试准备

## 候选人与岗位匹配摘要

## 本轮面试目标

## 建议时间分配

## 核心问题与追问

## 重点风险验证

## 评分建议

## 面试官记录区
```

## 文件抽取策略

skills 不强绑定某个平台的专属工具名，而是给 agent 明确的处理策略：

- Markdown 和 TXT：直接读取文本内容。
- PDF：优先使用当前 agent 环境可用的 PDF 文本抽取能力；若抽取文本明显为空、乱码或页数不完整，标记为需要人工复核。
- DOCX：优先使用当前 agent 环境可用的文档读取能力；若无法读取，提示用户转换为 PDF 或 TXT。
- 对所有简历文件，报告中记录文件名和解析状态。

首期不内置 OCR、向量库、数据库或外部 API，避免引入额外依赖和隐私风险。后续如团队需要处理扫描版 PDF，可单独扩展 OCR 方案。

## 安全与隐私

- 不在日志、注释或报告中输出密钥、token 或无关敏感信息。
- 报告只引用与招聘评估相关的必要证据摘要，避免完整复制简历原文。
- 不臆造候选人经历；没有证据的信息必须标记为“未在简历中找到”。
- 不基于年龄、性别、婚育、民族、宗教等无关或敏感属性评分。
- 如 JD 或简历中包含不合规筛选条件，报告应提示复核，而不是执行歧视性筛选。

## 错误处理

- JD 文件缺失或不可读：停止并要求用户提供有效 JD。
- 简历目录为空：停止并说明没有可评估文件。
- 单份简历解析失败：在报告中列为“解析失败”，不中断其他简历处理。
- 评分证据不足：降低置信度，并明确列出需要人工确认的信息。
- 输出路径未指定：使用 `recruit-assist-output/` 并生成清晰文件名。

## 验证方式

首期验证包含：

1. 插件 manifest 校验：确保 `.codex-plugin/plugin.json` 可通过 Codex 插件校验脚本。
2. Skill 基础校验：确保每个 `SKILL.md` 有有效 frontmatter，名称为 kebab-case，描述能覆盖触发场景。
3. 样例任务验证：使用一份样例 JD、一份样例 PDF 简历和一份样例 DOCX 简历，检查两个 skill 是否能生成结构完整的 Markdown。
4. 安全检查：确认文档中没有要求输出 token、密钥或基于敏感属性评分。

当前目录不是 git 仓库，因此设计文档不能在当前步骤提交。后续如果用户初始化或提供公司 Git 仓库，本设计文档应随插件源码一起提交。

## 非目标

首期不实现：

- 自动连接 ATS、邮箱、数据库或第三方招聘系统。
- 自动发送候选人邮件。
- 扫描版 PDF 的 OCR。
- 多语言翻译流水线。
- 长期候选人数据库或向量检索。
- Web UI 或后台服务。

这些能力可在插件首期稳定后作为后续迭代。

## 待用户确认的实现细节

实现前需要确认：

1. 公司或团队名称，用于 manifest 的 `author.name` 和界面展示。
2. 是否要使用默认 `MIT` license。
3. README 中是否只写本地安装说明，还是同时写公司 Git 仓库安装说明模板。
